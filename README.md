# FORGE -- Forward Reasoning Graph Engine

A physics solver that **derives answers through step-by-step forward chaining**
rather than recalling memorized patterns.

---

## Quick Start

```bash
pip install -r requirements.txt
python main.py
python tests/test_solver.py
```

---

## Solving Questions Interactively

A question bank lives in `data/questions/kinematics_questions.json`.
Pick a question and FORGE solves it, shows all derivation paths as a graph,
and saves a visual HTML diagram.

```bash
# interactive menu - pick a question
python solve_question.py

# or solve a specific question directly
python solve_question.py Q1

# solve all questions
python solve_question.py all
```

For each question FORGE prints:
  - all possible derivation paths (compact + detailed graph)
  - the selected shortest path
  - the wave-by-wave execution trace
  - the final answer

It also saves a visual graph to `output/Q1_graph.html` that you can
open in any browser. Each path is shown as a flow diagram from the
given variables through equations to the target.

To add your own question, add an entry to the questions JSON:

```json
{
  "id": "Q8",
  "title": "Find final velocity",
  "statement": "An object starts at 3 m/s and accelerates at 2 m/s^2 for 5 s. Find final velocity.",
  "given": {
    "/quant/physia/initial_velocity": 3.0,
    "/result/phy/acceleration": 2.0,
    "/quant/physia/time": 5.0
  },
  "target": "/result/phy/final_velocity_derived",
  "unit": "m/s"
}
```

## The Core Idea

When an LLM is asked "Given u=2, v=10, t=4, m=3 -- find Power", it retrieves
a cached answer from training data. It does not reason.

FORGE starts from the given variables, plans the shortest derivation path,
then executes it wave by wave -- stopping only when the target is derived
or when it honestly declares the information insufficient.

---

## Algorithm -- Two Phases

### Phase 1 -- PLAN

```
1. Build dependency graph from equation library
2. Find ALL derivation paths from G to target (backward DFS)
3. Rank paths by length (shortest first)
4. Print all paths to user
5. Select shortest path
```

### Phase 2 -- EXECUTE

```
Initialize:
    K0 = G                    (known set = all given variables)
    t  = 0

Loop:
    t.1  if target in Kt      -> RETURN answer       SUCCESS
    t.2  Et = { fi in F :
                  I(fi) in Kt (all inputs known)
               AND O(fi) not in Kt (output not yet produced) }
    t.3  if Et is empty       -> RETURN UNSOLVABLE   DEADLOCK
    t.4  execute all fi in Et (sorted by priority)
    t.5  K_{t+1} = Kt + outputs
         t = t + 1  -> goto t.1
```

## Protobuf Setup (sir's requirement)

The .proto files define the schema. To use binary protobuf loading:

```bash
# Step 1: Generate Python bindings
python -m grpc_tools.protoc -I src/protos --python_out=src/protos src/protos/funt/func.proto

# Step 2: Convert JSON to binary .pb
python scripts/json_to_proto.py

# Step 3: In main.py, switch to
fs = FunctionStore.from_proto("data/functions/physics_functions.pb")
```

Currently `from_json()` is used for readability. Both methods produce identical results.

## Adding a New Equation

1. Add entry to `data/functions/physics_functions.json`
2. Add elif branch to `src/engine/executor.py`
3. Done. Solver loop needs zero changes.

## Research Context

FORGE is the reasoning engine developed as part of **VLM-INSTINCT** research
at Jio Institute under Dr. Mohna Chakraborty.

The physics solver is the proof-of-concept. The same forward chaining algorithm
applies to visual question answering -- where the VLM handles perception and
FORGE handles structured reasoning over the extracted scene graph.
